# 依頼内容

現状の `docs/` ディレクトリの内容を確認し、開発タスクを整理してください。  
上位のタスク一覧を `task.md` に作成し、各タスクの詳細は `task/` フォルダに **1タスク=1ファイル** で作成します。ファイル名は **{タスク名}.md** とします。

# 手順

1. `docs/` 配下を読み込み、仕様・要件・未決事項・既存TODOを抽出する
2. 抽出結果から重複を統合し、実行可能なタスクに分解（粒度は1～2日で終わるレベル）
3. 依存関係・優先度を付与し、`task.md` を新規作成/更新
4. `task/` フォルダを作成（なければ）し、各タスクの詳細ファイル `{タスク名}.md` を作成
5. `task.md` から各詳細ファイルへのリンクを張る

# 出力要件

- `task.md` には以下を記載
  - 概要（このリストの目的）
  - タスクリスト（番号付き）
    - `優先度(P0/P1/P2)`、`担当(未定可)`、`見積り(人日)`、`依存先`、`ステータス(TODO/DOING/DONE)`
    - 各タスクの詳細ファイルへのリンク（例：`task/認証基盤整備.md`）
- 各 `task/{タスク名}.md` のテンプレート
  - タイトル: `{タスク名}`
  - 目的 / 背景（`docs/` のどの情報に基づくかを引用）
  - 完了条件（検証可能な受け入れ条件）
  - スコープ（含む/含まない）
  - 作業手順（チェックリスト）
  - 依存関係 / リスク
  - 見積り / 担当 / 期日
  - 参考リンク（`docs/` 内の該当ファイル・章への相対パス）

# ルール

- ファイル名は空白を許可（必要に応じて全角/半角は原文維持）。OS依存文字は使用しない。
- 既存の `task.md` / `task/` があれば追記・更新（重複は統合、履歴は残す）
- 可能な限り日本語で記載し、箇条書きを基本とする
- タスクは「成果物ベース」で定義（何ができるようになるかを明記）

# 例（`task.md` の1項目）

1. 認証基盤整備 — **P0** / 担当: 未定 / 見積り: 2
   - 依存: 設計方針決定
   - ステータス: TODO
   - 詳細: `task/認証基盤整備.md`

# 実行後の最終出力

- ルート: `task.md`
- ディレクトリ: `task/`（各タスクの `{タスク名}.md` が格納）
